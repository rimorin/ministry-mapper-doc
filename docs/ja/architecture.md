## データアーキテクチャ

### エンティティ関係モデル

システムは会衆レベルの分離を持つ階層的なデータ構造で組織されています：

```
Congregation（組織の境界）
    ├── Territory（地理的地域）
    │   ├── Map（建物/場所）
    │   │   └── Address（個別ユニット）
    │   └── Option（住所タイプ分類）
    ├── User（システムユーザー）
    │   └── Role（権限割り当て）
    ├── Assignment（伝道者アクセス）
    └── Message（通知）
```

### コアコレクション

**1. Congregations（会衆）** - 組織設定と境界
- マルチテナント分離を制御
- 割り当て期間と再試行制限を定義
- 地域設定（タイムゾーン、国）

**2. Territories（テリトリー）** - 地理的区分
- 複数のマップを含む
- 全体の完了率を追跡
- 奉仕地域を整理

**3. Maps（マップ）** - 特定の場所（建物）
- 単層または多層構造
- 地理座標
- 進捗追跡と統計

**4. Addresses（住所）** - 個別ユニット/世帯
- 状態追跡（未完了、完了、不在、DNC）
- 訪問メモと履歴
- 階と部屋情報

**5. Users（ユーザー）** - システムアカウント
- メール/パスワードまたはOAuth2認証
- 多要素認証サポート
- 会衆固有の役割

**6. Roles（役割）** - 権限割り当て
- 3レベル：read_only、conductor、administrator
- 会衆固有の権限
- 役割ベースのアクセス制御

**7. Assignments（割り当て）** - 伝道者アクセストークン
- 時間制限付きアクセスリンク
- 匿名フィールドワーカーアクセス
- 自動期限切れ

**8. Options（オプション）** - 住所分類
- 会衆ごとにカスタマイズ可能
- カウント可能/不可能なタイプ
- 進捗計算に影響

**9. Messages（メッセージ）** - コミュニケーション
- 伝道者からのフィードバック
- 管理者の指示
- メール通知

### 状態管理戦略

**4層アプローチ：**

1. **グローバルコンテキスト** - テーマ、言語、アプリモード
2. **カスタムフック** - ドメイン固有のロジックとデータ
3. **ローカルコンポーネント状態** - UIトグルとフォーム
4. **永続状態** - 設定用 LocalStorage

**メリット：**
- Redux のオーバーヘッドなし
- シンプルなデータフロー
- より良いコード分割
- テストが容易

### データフローパターン


## セキュリティアーキテクチャ

### 認証方式

1. **メール/パスワード** - bcryptハッシュを使用した標準認証
2. **OAuth2（Google）** - PKCEフローによるソーシャルログイン
3. **ワンタイムパスワード（OTP）** - 追加セキュリティレイヤー
4. **多要素認証（MFA）** - 管理者向けの強化されたセキュリティ

### 認可モデル

**役割ベースのアクセス制御（RBAC）：**

| 役割 | 権限 | ユースケース |
|------|-------------|----------|
| **read_only** | データ閲覧のみ | オブザーバー、監査者 |
| **conductor** | 割り当て作成、マップ管理 | フィールドコーディネーター |
| **administrator** | フルアクセス | システム管理者 |

### アクセス制御パターン

**パターン1：役割を持つ認証済みユーザー**
```
ユーザーがログインしている必要がある
+ ユーザーがrolesコレクションに役割を持つ
+ 役割が特定の会衆に対するもの
```

**パターン2：リンクベースアクセス**
```
割り当てIDがヘッダーで渡される
+ 割り当てが存在し期限切れでない
+ 割り当てが要求されたマップに属する
```

**パターン3：会衆の分離**
```
すべてのエンティティが会衆の外部キーを持つ
+ アクセスルールが会衆でフィルタリング
+ 会衆をまたいだデータアクセスを防止
```

## リアルタイム同期

### SSEアーキテクチャ

- **PocketBase リアルタイムAPI**：Server-Sent Events（SSE）
- **自動サブスクリプション**：コンポーネントがコレクションをサブスクライブ
- **ライブ更新**：変更がすべての接続クライアントにブロードキャスト
- **競合解決**：タイムスタンプベースの解決
- **可視性検出**：タブがアクティブになったときの自動更新

### サブスクリプションパターン

```typescript
// コンポーネントはデータ変更をサブスクライブする
useRealtimeSubscription('addresses', (data) => {
  // UIが自動的に更新される
});
```

## バックグラウンドジョブ

### スケジュールタスク

| ジョブ | スケジュール | 目的 |
|-----|----------|---------|
| **割り当てクリーンアップ** | 5分ごと | 期限切れ割り当ての削除 |
| **テリトリー集計** | 10分ごと | 進捗統計の更新 |
| **メッセージ処理** | 30分ごと | メッセージ通知の送信 |
| **指示** | 30分ごと | 管理者メッセージの送信 |
| **メモ更新** | 1時間ごと | メモ変更の通知 |
| **月次レポート** | 毎月1日 | Excelレポート生成（オプションのAI要約付き） |
| **未プロビジョニングユーザー** | 毎日 01:00 UTC | ユーザーライフサイクルの実行（警告→無効化→削除） |
| **非アクティブユーザー** | 毎日 01:30 UTC | 非アクティブアカウントの警告と無効化 |

### フィーチャーフラグ制御

すべてのバックグラウンドジョブは LaunchDarkly フィーチャーフラグで制御されます：
- デプロイなしでジョブを有効化/無効化
- 段階的ロールアウト機能
- 緊急停止スイッチ

### AI/LLM統合

Ministry Mapper は OpenAI の GPT モデルとインテグレーションして、インテリジェントなテキスト要約を実現します。これは `enable-report-ai-summary` LaunchDarkly フラグによってゲートされた**オプション**機能です。

**モデル**：GPT（`openai-go` SDK v3.29.0 経由で最新利用可能）  
**温度**：0.3（事実的で決定論的な出力）  
**レスポンス形式**：構造化データのための JSON モード

**ユースケース：**

| コンテキスト | ジョブ | 出力 |
|---------|-----|--------|
| 月次 Excel レポート | `generateMonthlyReport` | テリトリーごとの `summary` と `key_points` |
| 伝道者メッセージダイジェスト | `processMessages` | `overview` と `key_themes` |
| 会衆メモダイジェスト | `processNotes` | `overview` と `key_themes` |

**グレースフルデグラデーション**：`OPENAI_API_KEY` が設定されていない、または `enable-report-ai-summary` フラグがオフの場合、システムは AI 生成コンテンツなしのレポートとメールにフォールバックします—エラーは発生しません。

## スケーラビリティの考慮事項

### 現在の制限

- SQLite シングルライター制限
- 小〜中規模の会衆に適合（同時接続ユーザー1000人未満）
- 垂直スケーリング（より強力なサーバー）

### 将来のスケーラビリティ

- PocketBase は読み取りレプリカをサポート
- 必要に応じて PostgreSQL に移行可能
- 静的アセット用 CDN キャッシュ
- API 最適化のためのエッジ関数

## テクノロジーの決定

### なぜこれらのテクノロジーを選んだのか？

**React 19**：最新機能、自動最適化、小さなバンドル

**TypeScript**：型安全性、より良い IDE サポート、ランタイムエラーの削減

**Vite**：高速な開発サーバー、インスタント HMR、最適化されたビルド

**PocketBase**：セルフホスト、ベンダーロックインなし、メンテナンスが容易

**Wouter**：軽量（React Router の 40KB に対して 3KB）

**Bootstrap**：実績あり、包括的なコンポーネント、優れたアクセシビリティ

**Leaflet**：無料マップ、API制限なし、軽量

**SQLite**：自己完結型、別途データベースサーバー不要、セルフホストに最適
