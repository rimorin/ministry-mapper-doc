## 数据架构

### 实体关系模型

系统以层级方式组织数据，实现会众级别的隔离：

```
会众（组织边界）
    ├── 地区（地理区域）
    │   ├── 地图（建筑/位置）
    │   │   └── 地址（单个单元）
    │   └── 选项（地址类型分类）
    ├── 用户（系统用户）
    │   └── 角色（权限分配）
    ├── 分配（传道员访问）
    └── 消息（通知）
```

### 核心集合

**1. 会众** - 组织设置和边界
- 控制多租户隔离
- 定义分配时长和重试限制
- 区域设置（时区、国家）

**2. 地区** - 地理划分
- 包含多个地图
- 跟踪综合完成百分比
- 组织外勤服务区域

**3. 地图** - 具体位置（建筑）
- 单层或多层结构
- 地理坐标
- 进度跟踪和统计

**4. 地址** - 单个单元/住户
- 状态跟踪（未完成、已完成、无人在家、请勿打扰）
- 拜访备注和历史
- 楼层和单元信息

**5. 用户** - 系统账户
- 邮箱/密码或 OAuth2 身份验证
- 多因素身份验证支持
- 会众特定角色

**6. 角色** - 权限分配
- 三个级别：只读、组长、管理员
- 会众特定权限
- 基于角色的访问控制

**7. 分配** - 传道员访问令牌
- 限时访问链接
- 匿名外勤工作者访问
- 自动过期

**8. 选项** - 地址分类
- 每个会众可自定义
- 可计数与不可计数类型
- 影响进度计算

**9. 消息** - 通信
- 传道员反馈
- 管理员指令
- 邮件通知

```
  └── i18n 配置
```

### 状态管理策略

**四层方法：**

1. **全局上下文** - 主题、语言、应用模式
2. **自定义钩子** - 领域特定逻辑和数据
3. **本地组件状态** - UI 切换和表单
4. **持久化状态** - 偏好设置的 LocalStorage

**优势：**
- 无 Redux 开销
- 更简单的数据流
- 更好的代码拆分
- 更易于测试

### 数据流模式

```
用户操作
    ↓
组件事件处理程序
    ↓
自定义钩子（验证、状态管理）
    ↓
PocketBase 工具（CRUD 操作）
    ↓
PocketBase 后端（权限、数据库）
    ↓
实时订阅（SSE）
    ↓
UI 自动更新
```

## 安全架构

### 身份验证方式

1. **邮箱/密码** - 使用 bcrypt 哈希的标准身份验证
2. **OAuth2（Google）** - 带 PKCE 流的社交登录
3. **一次性密码（OTP）** - 额外安全层
4. **多因素身份验证（MFA）** - 管理员的增强安全性

### 授权模型

**基于角色的访问控制（RBAC）：**

| 角色 | 权限 | 使用场景 |
|------|------|------|
| **read_only** | 仅查看数据 | 观察员、审计员 |
| **conductor** | 创建分配、管理地图 | 外勤协调员 |
| **administrator** | 完全访问 | 系统管理员 |

### 访问控制模式

**模式 1：带角色的已认证用户**
```
用户必须已登录
+ 用户在角色集合中有角色
+ 角色针对特定会众
```

**模式 2：链接访问**
```
分配 ID 在请求头中传递
+ 分配存在且未过期
+ 分配属于请求的地图
```

**模式 3：会众隔离**
```
每个实体都有会众外键
+ 访问规则按会众筛选
+ 防止跨会众数据访问
```

## 后台任务

### 定时任务

| 任务 | 调度 | 标志 | 用途 |
|------|------|------|------|
| **分配清理** | 每 5 分钟 | `enable-assignments-cleanup` | 删除已过期分配 |
| **地区聚合** | 每 10 分钟 | `enable-territory-aggregations` | 更新进度统计 |
| **消息处理** | 每 30 分钟 | `enable-message-processing` | 发送消息通知 |
| **指令处理** | 每 30 分钟 | `enable-instruction-processing` | 发送管理员消息 |
| **备注更新** | 每 1 小时 | `enable-note-processing` | 通知备注变更 |
| **月度报告** | 每月 1 日 02:00 SGT | `enable-monthly-report` | 生成 Excel 报告（含可选 AI 摘要） |
| **未配置用户** | 每日 02:00 SGT | `enable-unprovisioned-user-processing` | 执行用户生命周期（警告→禁用→删除） |
| **不活跃用户** | 每日 02:30 SGT | `enable-inactive-user-processing` | 警告并禁用不活跃账户 |
| **新增地址通知** | 每日 03:00 SGT | `enable-new-addresses-notification` | **新增** — 应用内新增地址的每日摘要邮件 |

### 功能标志控制

所有后台任务由 LaunchDarkly 功能标志控制：
- 无需部署即可启用/禁用任务
- 逐步推出能力
- 紧急关闭开关

### AI/LLM 集成

Ministry Mapper 集成 OpenAI 的 GPT 模型用于智能文本摘要。这是一个由 `enable-report-ai-summary` LaunchDarkly 标志控制的**可选**功能。

**模型**：GPT（通过 `openai-go` SDK v3.29.0 使用最新版本）  
**温度**：0.3（事实性、确定性输出）  
**响应格式**：结构化数据的 JSON 模式

**使用场景：**

| 上下文 | 任务 | 输出 |
|--------|------|------|
| 月度 Excel 报告 | `generateMonthlyReport` | 每个地区的 `summary` 和 `key_points` |
| 传道员消息摘要 | `processMessages` | `overview` 和 `key_themes` |
| 会众备注摘要 | `processNotes` | `overview` 和 `key_themes` |

**优雅降级**：如果未设置 `OPENAI_API_KEY` 或 `enable-report-ai-summary` 标志关闭，系统回退到不含 AI 生成内容的报告和邮件——不会产生错误。

### 前端优化

- **代码拆分**：懒加载路由
- **打包优化**：摇树优化、压缩
- **React 19 编译器**：自动记忆化
- **PWA 缓存**：静态资源的 Service Worker
- **虚拟化列表**：大型数据集使用 React-window

### 后端优化

- **数据库索引**：25+ 索引加速查询
- **聚合缓存**：地区进度统计通过批量定时任务（`updateTerritoryAggregates`，每 10 分钟运行，11 分钟回溯窗口）预先计算，替代了之前的实时防抖方案。结果以 JSON 格式存储以便快速检索
- **连接池**：高效的数据库连接
- **事务批处理**：减少数据库往返次数

## 技术决策

### 为什么选择这些技术？

**React 19**：最新功能、自动优化、更小的打包

**TypeScript**：类型安全、更好的 IDE 支持、更少的运行时错误

**Vite**：快速开发服务器、即时 HMR、优化的构建

**PocketBase**：自托管、无供应商锁定、易于维护

**Wouter**：轻量级（3KB 对比 React Router 的 40KB）

**Bootstrap**：经过验证、全面的组件、良好的可访问性

**Leaflet**：免费地图、无 API 限制、轻量级

**SQLite**：自包含、无需单独的数据库服务器、非常适合自托管
