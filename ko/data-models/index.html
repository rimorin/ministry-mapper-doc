    데이터 모델 및 스키마¶

    개요¶

    Ministry Mapper는 멀티 테넌트 회중 관리를 위해 설계된 계층적 데이터 모델을 사용하여 PocketBase를 통해 SQLite를 사용합니다. 이 문서는 모든 데이터 모델, 관계 및 비즈니스 규칙에 대한 포괄적인 문서를 제공합니다.

    엔티티 관계 다이어그램¶

    회중 (루트 엔티티 - 멀티 테넌트 격리)
    │
    ├─── 구역 (1:M)
    │      │
    │      └─── 지도 (1:M)
    │             │
    │             ├─── 주소 (1:M)
    │             │      ├─── addresses_log (1:M)
    │             │      └─── address_options (1:M)
    │             ├─── 배정 (1:M)
    │             └─── 메시지 (1:M)
    │
    ├─── 옵션 (1:M)
    │      └─── address_options (주소와의 M:M 연결 테이블)
    │
    ├─── 사용자 (역할을 통해 1:M)
    │      │
    │      ├─── 역할 (1:M)
    │      └─── 배정 (1:M)
    │
    ├─── 메시지 (1:M)
    │
    └─── aggregates (구역 및 지도에서 1:M)

    핵심 컬렉션¶

    1. 회중¶

    목적: 멀티 테넌트 격리 및 조직 설정을 위한 중앙 엔티티

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    code text 예 조직 코드
    name text 예 조직 이름
    expiry_hours number 예 배정 링크 기간 (시간)
    max_tries number 예 완료로 표시되기 전 최대 "부재" 시도 횟수
    origin select 예 국가/지역 (sg, my)
    timezone select 예 날짜 계산을 위한 타임존
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    접근 규칙: - 목록: 사용자는 회중에 역할이 있어야 함 - 보기: 역할이 있는 사용자 또는 링크 기반 접근 - 업데이트: 관리자만 - 삭제: 관리자만

    비즈니스 규칙: - 모든 지도의 기본 배정 기간 제어 - max_tries는 "부재" 주소가 완료로 계산되는 시점 결정 - 타임존은 날짜 기반 계산 및 보고서에 영향

    예시: 

    {
  "id": "abc123",
  "code": "CONG001",
  "name": "북부 회중",
  "expiry_hours": 24,
  "max_tries": 3,
  "origin": "sg",
  "timezone": "Asia/Singapore"
}

    2. 구역¶

    목적: 전도 서비스 구성을 위한 회중 내 지리적 구분

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    congregation relation 예 회중에 대한 FK (연쇄 삭제)
    code text 예 구역 식별자 (자동: "0")
    description text 아니오 사람이 읽을 수 있는 설명
    progress number 자동 완료 백분율 (0-100)
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    인덱스: - (congregation, code) - 회중별 빠른 조회 - (congregation) - 회중 필터

    관계: - 지도와 1:M - 주소와 1:M (성능을 위해 역정규화)

    접근 규칙: - 목록: 회중에 역할이 있는 사용자 - 보기: 역할이 있는 사용자 또는 링크 접근 - 생성/업데이트/삭제: 인도자 또는 관리자

    진행 상황 계산: 구역 내 모든 지도의 집계에서 자동 계산

    예시: 

    {
  "id": "terr_abc",
  "congregation": "cong_123",
  "code": "1A",
  "description": "시내 지구",
  "progress": 65.5
}

    3. 지도¶

    목적: 주소를 포함하는 특정 위치 또는 건물을 나타냄

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    territory relation 예 구역에 대한 FK (연쇄 삭제)
    congregation relation 예 회중에 대한 FK (연쇄 삭제)
    sequence number 자동 구역 내 순서 (자동 증가)
    code text 예 지도 식별자/번호 (자동: "0")
    description text 아니오 위치 설명
    type select 예 single 또는 multi (층 유형)
    coordinates JSON 아니오 지리적 위치 {lat, lng}
    aggregates JSON 자동 캐시된 통계 (최대 2MB)
    progress number 자동 완료 백분율
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    인덱스: - (territory, code) - 기본 조회 - (territory, sequence) - 정렬 - (territory) - 구역 필터

    연쇄 삭제: - 주소 - 배정 - 메시지

    집계 구조: 

    {
  "done": 45,
  "not_done": 30,
  "dnc": 5,
  "invalid": 2,
  "not_home_max_tries": 8
}

    지도 유형: - single: 개인 주소 (집) - multi: 공공 주소 (다층 건물)

    접근 규칙: - 목록: 역할이 있는 사용자 또는 링크 접근 - 보기: 역할이 있는 사용자 또는 링크 접근 - 생성/업데이트: 인도자 또는 관리자 - 삭제: 관리자만

    4. 주소¶

    목적: 지도 내 개별 단위 또는 가구

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    map relation 예 지도에 대한 FK (연쇄 삭제)
    congregation relation 예 회중에 대한 FK (역정규화)
    territory relation 예 구역에 대한 FK (역정규화)
    floor number 예 층 번호 (0 기반)
    code text 예 단위/문 코드 (예: "101", "A")
    sequence number 예 지도 내 순서
    status select 예 주소 방문 상태
    not_home_tries number 자동 "부재" 시도 카운터
    type relation 아니오 옵션에 대한 M:M (JSON 배열)
    notes text 아니오 현장 작업자 메모
    last_notes_updated date 자동 메모가 마지막으로 변경된 시점 (훅을 통해)
    last_notes_updated_by text 자동 마지막 업데이터의 사용자 이름 (훅을 통해)
    dnc_time date 아니오 "방문 거부"로 표시된 시점
    coordinates JSON 아니오 세밀한 GPS {lat, lng}
    updated_by text 아니오 업데이트한 사람 추적
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    상태 값: - not_done - 아직 방문하지 않음 (초기 상태) - done - 성공적으로 완료 - not_home - 아무도 없음 (재시도 가능) - do_not_call - 작업에서 제외 - invalid - 주소가 존재하지 않음

    인덱스 (7개): - (map) - 지도 필터 - (map, status) - 상태 분포 쿼리 - (map, code) - 코드 조회 - (map, floor) - 층 구성 - (type, congregation) - 옵션 분포 - (congregation, last_notes_updated_by) - 사용자별 최근 메모 - (territory, status) - 구역 전체 상태

    자동 훅: - 업데이트는 last_notes_updated 및 last_notes_updated_by 필드를 업데이트 - 지도에서 집계 재계산 트리거 - 구역 진행 상황 업데이트

    접근 규칙: - 목록: 역할이 있는 사용자 또는 링크 접근 - 보기: 역할이 있는 사용자 또는 링크 접근 - 생성: 인도자 또는 관리자 - 업데이트: 역할이 있는 사용자 또는 링크 접근 (전도인 업데이트 가능) - 삭제: 관리자만

    5. 사용자¶

    목적: 인증을 사용하는 시스템 사용자 계정

    유형: Auth 컬렉션 (PocketBase user-auth-type)

    시스템 필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    email email 예 로그인 ID (고유)
    emailVisibility bool 예 이메일 공개 여부
    verified bool 자동 이메일 인증 상태
    password password 예 Bcrypt 해싱 (비용: 10, 최소 6자)
    tokenKey text 숨김 토큰 생성 키 (30-60자)

    커스텀 필드:

    필드 유형 필수 설명
    name text 예 표시 이름 (2-50자)
    disabled bool 아니오 계정 비활성화 플래그
    last_login date 자동 마지막 인증 타임스탬프 (훅을 통해)

    이름 유효성 검사 패턴: ^[A-Za-z][\w\s\.\-']*$ (문자로 시작, 단어 문자, 공백, 점, 하이픈, 아포스트로피 허용)

    인증 구성: - 비밀번호 인증: 활성화 (최소 6자) - OAuth2: Google PKCE 흐름 - MFA: 활성화 (기간: 1800s) - OTP: 활성화 (기간: 180s, 길이: 4자리) - 이메일 인증: 기간 604800s (7일) - 비밀번호 재설정: 기간 1800s (30분)

    접근 규칙: - 목록: 관리자만 - 보기: 관리자 또는 자신 - 생성: 공개 (OAuth 또는 명시적 등록) - 업데이트: 자신만 - 삭제: 관리자만

    6. 역할¶

    목적: 사용자를 회중에 연결하는 권한 배정

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    user relation 예 사용자에 대한 FK (연쇄 삭제)
    congregation relation 예 회중에 대한 FK (연쇄 삭제)
    role select 예 권한 수준
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    역할 값: - read_only - 보기 전용 접근 - conductor - 배정 및 지도 관리 가능 - administrator - 전체 접근

    접근 규칙: - 생성/업데이트/삭제: 관리자만 - 목록: 자신의 역할 - 보기: 자신의 역할

    비즈니스 규칙: - 사용자당 회중당 하나의 역할 - 역할이 모든 행동에 대한 권한 결정

    7. 배정¶

    목적: 시간 제한 지도 접근을 위한 전도인 접근 토큰

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(20) 예 익명 접근 토큰으로 사용됨
    congregation relation 예 회중에 대한 FK (연쇄 삭제)
    map relation 예 지도에 대한 FK (연쇄 삭제)
    user relation 아니오 전도인 사용자 (선택)
    type select 예 배정 분류
    expiry_date date 예 자동 만료 타임스탬프
    publisher text 아니오 전도인 이름/식별자
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    배정 유형: - normal - 일반 구역 작업 - personal - 개인 전도 구역

    워크플로우: 1. 인도자가 /territory/link 엔드포인트를 통해 생성 2. 전도인이 20자 link_id(이 배정의 ID)를 받음 3. 전도인이 HTTP 헤더 @request.headers.link_id를 통해 link_id 전달 4. PocketBase가 유효성 검사: link_id가 assignment.id와 일치 AND expiry_date > now 5. 백그라운드 작업이 매 5분마다 만료된 배정 삭제

    8. 옵션¶

    목적: 주소 유형 분류 (회중별 맞춤화 가능)

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    congregation relation 예 회중에 대한 FK (연쇄 삭제)
    code text 예 옵션 식별자 (예: "RES", "BUS")
    description text 예 사람이 읽을 수 있는 레이블
    is_countable bool 예 진행 상황 계산에 포함
    is_default bool 예 새 주소의 기본 옵션
    sequence number 예 표시/정렬 순서
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    진행 상황에 미치는 영향: - 계산 가능 옵션 (is_countable=true)이 진행 상황 % 계산에 포함 - 계산 불가 옵션은 별도로 추적되지만 진행 상황에서 제외 - 옵션이 삭제되면 주소가 자동으로 기본 옵션을 받음

    9. 메시지¶

    목적: 사용자 간 커뮤니케이션 및 알림

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    map relation 예 지도에 대한 FK (연쇄 삭제)
    congregation relation 예 회중에 대한 FK (연쇄 삭제)
    message text 예 메시지 내용
    created_by text 예 작성자의 사용자 이름
    read bool 예 관리자가 읽었는지 여부
    pinned bool 예 관리자 고정 지시사항 플래그
    type select 예 수신자 유형
    created date 자동 생성 타임스탬프
    updated date 자동 마지막 업데이트 타임스탬프

    메시지 유형: - publisher - 전도인에서 관리자로 - conductor - 인도자로부터 - administrator - 관리자 지시사항 (고정 가능)

    백그라운드 처리: - 매 30분: 읽지 않은 메시지가 이메일을 통해 관리자에게 전송 - 매 30분: 고정된 관리자 메시지가 전도인에게 이메일로 전송 - 매 1시간: 메모 업데이트가 관리자에게 전송

    10. addresses_log¶

    목적: 주소 상태 변경의 감사 추적 — 주소 상태가 업데이트될 때마다 여기에 기록이 작성됩니다

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    address relation 예 주소에 대한 FK (연쇄 삭제)
    old_status text 예 이전 주소 상태 값
    new_status text 예 새 주소 상태 값
    changed_by relation 아니오 사용자에 대한 FK — 변경한 사용자
    created date 자동 변경 타임스탬프

    접근 규칙: - 목록/보기: 인도자 또는 관리자 - 생성: 시스템 훅 (주소 상태 변경 시 자동) - 업데이트/삭제: 관리자만

    11. aggregates¶

    목적: 구역 및 지도의 캐시된 진행 스냅샷. 10분마다 백그라운드 작업으로 재계산됩니다

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    notDone number 예 아직 방문하지 않은 주소 수
    notHome number 예 "부재" 주소 수
    progress number 예 완료 백분율 (0–100)
    territory relation 아니오 구역에 대한 FK — 구역 수준 집계에 설정
    map relation 아니오 지도에 대한 FK — 지도 수준 집계에 설정

    비즈니스 규칙: - 레코드당 territory 또는 map 중 정확히 하나가 설정됨 - 백그라운드 작업으로 10분마다 재계산 - 값은 읽기 전용 스냅샷; 실시간 진행 상황은 필요 시 계산

    접근 규칙: - 목록/보기: 역할이 있는 사용자 또는 링크 접근 - 생성/업데이트: 시스템 백그라운드 작업만 - 삭제: 관리자만

    12. address_options¶

    목적: 개별 주소에 적용된 주소 수준 옵션(options 컬렉션)을 추적 — 주소 ↔ 옵션 다대다 관계를 위한 연결 테이블

    유형: 기본 컬렉션

    필드:

    필드 유형 필수 설명
    id text(15) 예 자동 생성된 고유 ID
    address relation 예 주소에 대한 FK (연쇄 삭제)
    option relation 예 옵션에 대한 FK (연쇄 삭제)
    congregation relation 예 회중에 대한 FK (빠른 쿼리를 위한 비정규화)

    인덱스: - (address, option) — 고유 쌍 (주소당 하나의 옵션) - (congregation) — 회중 전체 옵션 사용

    접근 규칙: - 목록/보기: 역할이 있는 사용자 또는 링크 접근 - 생성/삭제: 인도자 또는 관리자

    분석 컬렉션¶

    분석 컬렉션은 보고 및 인사이트를 위한 이력 데이터를 저장합니다. 이 컬렉션은 내부적으로 관리되며 정확한 필드는 시간이 지남에 따라 변경될 수 있습니다. 개별 필드 테이블 대신 목적 위주로 문서화되어 있습니다.

    analytics_territories¶

    목적: 시간 경과에 따른 구역 수준 지표(예: 진행 백분율, 주소 수)의 주기적 스냅샷을 저장합니다. 이력 트렌드 차트 및 구역 완료 보고서 생성에 사용됩니다.

    analytics_maps¶

    목적: 시간 경과에 따른 지도 수준 지표의 주기적 스냅샷을 저장합니다. 개별 지도 완료율 및 활동 패턴에 대한 세부 보고를 가능하게 합니다.

    analytics_daily_status¶

    목적: 회중 전체의 주소 상태(완료, 미완료, 부재, 방문 금지, 유효하지 않음)에 대한 일별 집계 수를 저장합니다. 일별 및 주별 트렌드 대시보드를 지원합니다.

    analytics_not_home¶

    목적: 반복적으로 "부재"로 표시된 주소를 추적합니다. 인도자를 위한 후속 대상을 표시하고 지속적으로 연락이 닿지 않는 주소를 식별하는 데 사용됩니다.

    analytics_user_audit¶

    목적: 책임 및 감사 목적으로 로그인, 주소 업데이트, 배정 작업을 포함한 사용자 행동을 기록합니다. 관리자가 활동 이력을 검토하고 이상 징후를 조사할 수 있게 합니다.

    참고: 분석 컬렉션은 백그라운드 작업으로 채워지고 보고 레이어에서 읽습니다. 애플리케이션 코드에서의 직접 쓰기는 피해야 합니다.

    데이터베이스 기술¶

    엔진: PocketBase를 통한 SQLite (modernc.org/sqlite v1.40.2+)

    기능: - 순수 Go 구현 (CGo 없음) - 크로스 플랫폼 컴파일 가능 - ccgo/v4를 사용하여 Go로 변환된 C SQLite 기반 - 정기적인 보안 업데이트와 함께 적극적으로 유지 관리 (SQLite 3.50+ 정렬)

    사용된 SQL 기능: - 유연한 데이터를 위한 JSON 필드 (좌표, 집계) - 쿼리 최적화를 위한 복합 인덱스 - ACID 준수를 위한 트랜잭션 - 복잡한 집계를 위한 CTE (공통 테이블 표현식)

    주요 관계¶

    관계 기수성 연쇄 동작
    회중 → 구역 1:M 연쇄 삭제
    구역 → 지도 1:M 연쇄 삭제
    지도 → 주소 1:M 연쇄 삭제
    지도 → 배정 1:M 연쇄 삭제
    지도 → 메시지 1:M 연쇄 삭제
    주소 → 옵션 M:M address_options 연결 테이블을 통해
    사용자 → 역할 M:M 역할 테이블을 통해
    사용자 → 배정 1:M 배정 유지
    회중 → 옵션 1:M 연쇄 삭제
    회중 → 역할 M:M 역할 테이블을 통해
    주소 → addresses_log 1:M 연쇄 삭제
    구역 → aggregates 1:M 백그라운드 작업으로 재계산
    지도 → aggregates 1:M 백그라운드 작업으로 재계산
    주소 → address_options 1:M 연쇄 삭제
    옵션 → address_options 1:M 연쇄 삭제

    비즈니스 규칙¶

    진행 상황 계산¶

    공식: 

    completed = (done + not_home_max_tries) / total_countable * 100

    계산 가능한 주소: - is_countable=true 옵션을 가진 주소만 - DNC 및 유효하지 않은 주소 제외

    완료 기준: - 상태 = done, 또는 - 상태 = not_home AND not_home_tries >= congregation.max_tries

    주소 수명 주기¶

    not_done → not_home (재시도 시도) → done
        → do_not_call (제외)
        → invalid (제외)

    4. 집계 재계산 트리거

    8. 주소가 변경될 때 집계 재계산

    9. 구역 진행 상황 업데이트

    10. 배정 생성:

    11. 회중 설정에 따라 만료 설정
