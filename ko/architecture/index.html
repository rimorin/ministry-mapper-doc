    시스템 아키텍처¶

    데이터 아키텍처¶

    엔티티 관계 모델¶

    시스템은 회중 수준 격리로 계층 구조로 데이터를 구성합니다:

    회중 (조직 경계)
    ├── 구역 (지리적 지역)
    │   ├── 지도 (건물/위치)
    │   │   └── 주소 (개별 단위)
    │   └── 옵션 (주소 유형 분류)
    ├── 사용자 (시스템 사용자)
    │   └── 역할 (권한 배정)
    ├── 배정 (전도인 접근)
    └── 메시지 (알림)

    핵심 컬렉션¶

    1. 회중 - 조직 설정 및 경계 - 멀티 테넌트 격리 제어 - 배정 기간 및 재시도 제한 정의 - 지역 설정 (타임존, 국가)

    2. 구역 - 지리적 구분 - 여러 지도를 포함 - 집계 완료 백분율 추적 - 전도 서비스 영역 구성

    3. 지도 - 특정 위치 (건물) - 단층 또는 다층 구조 - 지리적 좌표 - 진행 상황 추적 및 통계

    4. 주소 - 개별 단위/가구 - 상태 추적 (미완료, 완료, 부재, 방문 거부) - 방문 메모 및 이력 - 층수 및 단위 정보

    5. 사용자 - 시스템 계정 - 이메일/비밀번호 또는 OAuth2 인증 - 다중 인증 지원 - 회중별 역할

    6. 역할 - 권한 배정 - 세 가지 수준: read_only, conductor, administrator - 회중별 권한 - 역할 기반 접근 제어

    7. 배정 - 전도인 접근 토큰 - 시간 제한 접근 링크 - 익명 현장 작업자 접근 - 자동 만료

    8. 옵션 - 주소 분류 - 회중별 맞춤화 가능 - 계산 가능 vs 계산 불가 유형 - 진행 상황 계산에 영향

    9. 메시지 - 커뮤니케이션 - 전도인 피드백 - 관리자 지시사항 - 이메일 알림

    프론트엔드 아키텍처¶

    컴포넌트 구성 패턴¶

    Ministry Mapper는 기능 기반 컴포넌트 구성을 사용합니다:

    페이지 (라우트)
  ├── 관리자 대시보드 (/)
  ├── 전도인 지도 (/map/:id)
  ├── 사용자 관리 (/usermgmt)
  └── 프론트 페이지 (/)

컴포넌트 (재사용 가능한 UI)
  ├── 폼 컴포넌트 (14개 컴포넌트)
  ├── 모달 컴포넌트 (24개 이상 모달)
  ├── 내비게이션 컴포넌트 (28개 컴포넌트)
  ├── 테이블 컴포넌트 (9개 컴포넌트)
  ├── 지도 컴포넌트 (7개 컴포넌트)
  ├── 미들웨어/프로바이더 (5개 컴포넌트)
  └── 정적 페이지 (8개 컴포넌트)

비즈니스 로직 (커스텀 훅)
  ├── useTerritoryManagement
  ├── useMapManagement
  ├── useCongManagement
  ├── useModalManagement
  ├── useRealtimeSubscription
  └── [4개 이상...]

유틸리티 및 타입
  ├── TypeScript 인터페이스
  ├── 상수
  ├── 헬퍼 함수 (30개 이상)
  ├── PocketBase 통합
  ├── 권한 부여 정책
  └── i18n 구성

    상태 관리 전략¶

    4계층 접근법:

    1. 전역 컨텍스트 - 테마, 언어, 앱 모드
    2. 커스텀 훅 - 도메인별 로직 및 데이터
    3. 로컬 컴포넌트 상태 - UI 토글 및 폼
    4. 영속 상태 - 기환경설정을 위한 LocalStorage

    이점: - Redux 오버헤드 없음 - 더 단순한 데이터 흐름 - 더 나은 코드 분리 - 더 쉬운 테스팅

    데이터 흐름 패턴¶

    사용자 동작
    ↓
컴포넌트 이벤트 핸들러
    ↓
커스텀 훅 (유효성 검사, 상태 관리)
    ↓
PocketBase 유틸리티 (CRUD 작업)
    ↓
PocketBase 백엔드 (권한, 데이터베이스)
    ↓
실시간 구독 (SSE)
    ↓
UI 자동 업데이트

    주요 설계 패턴¶

    패턴 사용 목적
    미들웨어 래퍼 Sentry 통합 중앙화된 오류 추적
    이벤트 훅 PocketBase 수명 주기 자동 업데이트
    작업 스케줄러 기능 플래그가 있는 Cron 예약된 작업
    트랜잭션 패턴 데이터베이스 작업 데이터 일관성
    링크 기반 접근 배정 토큰 익명 접근
    집계 캐싱 배치 cron (10분 / 11분 룩백) 사전 계산된 구역 진행 통계
    커스텀 훅 비즈니스 로직 재사용 가능한 로직
    프로바이더 패턴 컨텍스트 프로바이더 전역 상태
    지연 로딩 라우트 분리 성능
    정책 패턴 권한 부여 보안 규칙

    보안 아키텍처¶

    인증 방법¶

    1. 이메일/비밀번호 - bcrypt 해싱을 사용한 표준 인증
    2. OAuth2 (Google) - PKCE 흐름을 사용한 소셜 로그인
    3. 일회용 비밀번호 (OTP) - 추가 보안 계층
    4. 다중 인증 (MFA) - 관리자를 위한 향상된 보안

    권한 부여 모델¶

    역할 기반 접근 제어 (RBAC):

    역할 권한 사용 사례
    read_only 데이터 보기만 가능 관찰자, 감사자
    conductor 배정 만들기, 지도 관리 현장 담당자
    administrator 전체 접근 시스템 관리자

    접근 제어 패턴¶

    패턴 1: 역할이 있는 인증된 사용자 

    사용자가 로그인해야 함
+ 사용자는 역할 컬렉션에 역할을 가짐
+ 역할은 특정 회중을 위한 것

    패턴 2: 링크 기반 접근 

    헤더에 배정 ID 전달
+ 배정이 존재하고 만료되지 않음
+ 배정이 요청된 지도에 속함

    패턴 3: 회중 격리 

    모든 엔티티는 회중 외래 키를 가짐
+ 접근 규칙이 회중별로 필터링
+ 회중 간 데이터 접근 방지

    실시간 동기화¶

    SSE 아키텍처¶

    • PocketBase 실시간 API: 서버 전송 이벤트 (SSE)
    • 자동 구독: 컴포넌트가 컬렉션에 구독
    • 실시간 업데이트: 연결된 모든 클라이언트에 변경 사항 전파
    • 충돌 해결: 타임스탬프 기반 해결
    • 가시성 감지: 탭이 활성화될 때 자동 새로고침

    구독 패턴¶

    // 컴포넌트가 데이터 변경에 구독
useRealtimeSubscription('addresses', (data) => {
  // UI가 자동으로 업데이트
});

    예약된 작업¶

    작업 스케줄 플래그 목적
    cleanUpAssignments 매 5분 enable-assignments-cleanup 만료된 배정 삭제
    updateTerritoryAggregates 매 10분 enable-territory-aggregations 구역 진행 상황 통계 재계산
    processMessages 매 30분 enable-message-processing 전도인 메시지 알림 전송
    processInstructions 매 30분 enable-instruction-processing 관리자 메시지 전송
    processNotes 매 1시간 enable-note-processing 메모 변경 알림
    generateMonthlyReport 매월 1일 02:00 SGT enable-monthly-report Excel 보고서 생성 (AI 요약 선택 가능)
    processUnprovisionedUsers 매일 02:00 SGT enable-unprovisioned-user-processing 역할 없는 사용자 경고/비활성화

    processNewAddresses 매일 03:00 SGT enable-new-addresses-notification 앱에서 생성된 주소에 대한 일일 요약 이메일

    기능 플래그 제어¶

    모든 백그라운드 작업은 LaunchDarkly 기능 플래그로 제어됩니다: - 배포 없이 작업 활성화/비활성화 - 점진적 출시 기능 - 긴급 비활성화 스위치

    AI/LLM 통합¶

    Ministry Mapper는 지능형 텍스트 요약을 위해 OpenAI의 GPT 모델과 통합됩니다. 이것은 enable-report-ai-summary LaunchDarkly 플래그로 제어되는 선택적 기능입니다.

    모델: GPT (최신 openai-go SDK v3.29.0을 통해 사용 가능)
    온도: 0.3 (사실적, 결정론적 출력)
    응답 형식: 구조화된 데이터를 위한 JSON 모드

    사용 사례:

    컨텍스트 작업 출력
    월간 Excel 보고서 generateMonthlyReport 구역별 summary 및 key_points
    전도인 메시지 다이제스트 processMessages overview 및 key_themes
    회중 메모 다이제스트 processNotes overview 및 key_themes

    점진적 저하: OPENAI_API_KEY가 설정되지 않거나 enable-report-ai-summary 플래그가 꺼져 있으면, 시스템은 AI 생성 콘텐츠 없이 보고서 및 이메일로 폴백합니다. 오류가 발생하지 않습니다.

    프론트엔드 최적화¶

    • 코드 분리: 지연 로딩 라우트
    • 번들 최적화: 트리 쉐이킹, 최소화
    • React 19 컴파일러: 자동 메모이제이션
    • PWA 캐싱: 정적 에셋을 위한 서비스 워커
    • 가상화된 목록: 대용량 데이터셋을 위한 React-window

    백엔드 최적화¶

    • 데이터베이스 인덱스: 빠른 쿼리를 위한 25개 이상 인덱스
    • 집계 캐싱: 구역 진행 통계는 배치 cron 작업(updateTerritoryAggregates, 11분 룩백 윈도우로 10분마다)으로 사전 계산되며, 이전의 실시간 디바운서 방식을 대체합니다. 결과는 빠른 조회를 위해 JSON으로 저장됩니다.
    • 연결 풀링: 효율적인 데이터베이스 연결
    • 트랜잭션 일괄 처리: 데이터베이스 라운드 트립 감소

    기술 결정¶

    왜 이러한 기술을 선택했습니까?¶

    React 19: 최신 기능, 자동 최적화, 더 작은 번들

    TypeScript: 타입 안전성, 더 나은 IDE 지원, 더 적은 런타임 오류

    Vite: 빠른 개발 서버, 즉각적인 HMR, 최적화된 빌드

    PocketBase: 자체 호스팅, 벤더 종속 없음, 유지 관리 용이

    Wouter: 경량 (React Router의 40KB 대비 3KB)

    Bootstrap: 검증된, 포괄적인 컴포넌트, 좋은 접근성

    Leaflet: 무료 지도, API 제한 없음, 경량

    SQLite: 자체 포함, 별도 데이터베이스 서버 불필요, 셀프 호스팅에 완벽
